# 모아보기 기간 체계 재설계

## 목표

-   홈 `공감글`과 `/explore`의 기간 감각을 최대한 맞춘다.
-   프론트 탭 변경만 하지 않고 `explore.json` 생성 기준까지 함께 바꾼다.
-   기존 `24h / 7d / 30d` 상위 집계는 제거가 아니라 하위 호환 경로를 둔다.

## 현재 상태

-   홈 공감글은 `1h / 3h / 6h / 12h / 24h / 48h` 기반이다.
-   `/explore`의 `top` 모드는 `24h / 7d / 30d` 기반 `periods`와 `comment_periods`를 사용한다.
-   `hot`, `new`, `rising`은 단일 리스트이며 기간 파라미터가 없다.
-   프론트는 [`/home/angple/web/apps/web/src/lib/api/types.ts`](/home/angple/web/apps/web/src/lib/api/types.ts)에 정의된 `ExploreTopPeriod = '24h' | '7d' | '30d'`를 전제로 동작한다.

## 권장 방향

-   1차 통일 기간: `6h / 12h / 24h / 48h`
-   유지 이유:
    -   홈 공감글과 직접 대응된다.
    -   `1h / 3h`는 탐색 지표 변동성이 너무 커서 `/explore` 전체 페이지 기본 필터로는 소음이 크다.
    -   `7d / 30d`는 별도 `장기` 뷰나 보조 탭으로 남기는 편이 적절하다.

## 제안 스키마

기존:

```json
{
    "modes": {
        "top": {
            "periods": {
                "24h": [],
                "7d": [],
                "30d": []
            },
            "comment_periods": {
                "24h": [],
                "7d": [],
                "30d": []
            }
        }
    }
}
```

변경:

```json
{
    "schema_version": 2,
    "modes": {
        "top": {
            "periods": {
                "6h": [],
                "12h": [],
                "24h": [],
                "48h": []
            },
            "comment_periods": {
                "6h": [],
                "12h": [],
                "24h": [],
                "48h": []
            },
            "legacy_periods": {
                "7d": [],
                "30d": []
            },
            "legacy_comment_periods": {
                "7d": [],
                "30d": []
            }
        }
    }
}
```

## 집계 기준

-   `6h`: 최근 6시간 생성 글/댓글만 대상
-   `12h`: 최근 12시간 생성 글/댓글만 대상
-   `24h`: 최근 24시간 생성 글/댓글만 대상
-   `48h`: 최근 48시간 생성 글/댓글만 대상

정렬은 기존 `top_score`를 유지하되, 기간별 모집단을 분리해서 계산한다.

## 프론트 변경 순서

1. 타입 확장
2. 프론트에서 `schema_version` 또는 period key 존재 여부로 새 스키마 감지
3. 새 스키마가 있으면 `6h / 12h / 24h / 48h` 탭 노출
4. 없으면 기존 `24h / 7d / 30d` 탭 유지

## 백엔드/생성기 변경 순서

1. `explore.json` 생성 로직에 새 기간 집계 추가
2. 일정 기간 `legacy_periods` 동시 제공
3. 프론트 배포 후 구 스키마 제거

## 주의점

-   이 작업은 프론트 단독 변경으로 끝나지 않는다.
-   현재 웹 앱은 캐시 파일 로더만 가지고 있고 실제 집계는 외부 생성기에서 만들어진 JSON에 의존한다.
-   따라서 이번 UI 수정과 별도 배포 단위로 처리하는 것이 안전하다.
